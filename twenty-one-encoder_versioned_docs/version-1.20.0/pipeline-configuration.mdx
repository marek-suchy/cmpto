## 3.2 SMPTE 2022-7 input

Using RTP, it’s possible to configure redundant input paths (SMPTE 2022-7) to avoid potential data loss on unreliable networks or long-distance content transfers. To configure redundant paths, enable the **SMPTE 2022-7 Redundancy** checkbox and fill in the specific NIC, IP address, SSM (Source-Specific Multicast), and ports for both the primary and redundant paths. The primary path is always preferred in the event of a data mismatch between the paths.

Based on your network environment, you can configure the **Jitter buffer length** (expressed as a skew or difference between both paths in ms). This value should account for the maximum expected jitter packets from both paths might experience, including a safety headroom. The default value is 10 ms, which is sufficient for LAN and inter-facility links. For long-haul links or environments with higher relative delay, values up to approximately 100 ms may be used. The allowed range is 1 to 400 ms. Increasing this value improves tolerance to delay variation but increases overall processing latency.

## 3.3 MPTS for choosing the right program

MPTS (MPEG transport stream) is used in broadcast systems and is designed to be used in terrestrial or satellite broadcasts, for example. It can carry multiple programs and each program contains a pack of video streams, audio streams and metadata (if present).

Comprimato Twenty-One Encoder has the support for receiving and processing the MPTS streams and detects all programs and allows its users to choose them easily.

*Figure: CONFIGURING MPTS*

When you wish to choose which program to use, you need to allow the Program number configuration in the Input stream configuration options. Once allowed, the user can select the program number from a dropdown menu. The current program informs about the current configuration, while the Configured program has to be applied first to take effect. If you will not set up anything, the first program in the list is selected. If a Configured program number is not present in the stream, Twenty-one encoder reports an error. Once you change the program number and apply the configuration, a video with another PID is detected (if present).

## 3.4 Auto reconfiguration

Twenty-One Encoder allows the auto reconfiguration of the pipelines. It means it can automatically restart a pipeline when it detects a change in the parameters of the input video or audio. The video outputs are kept unchanged, the behaviour of audio outputs is described later. The auto reconfiguration can be turned on by a checkbox. This option is not available with ST 2110 input and will instead be controlled by NMOS IS-05 (Figure 26)

## 3.5 Auto reconfiguration of Audio outputs

There are three modes of output audio channel layout. Passthrough, Pairs and Quads. They can be selected in the dialog window, which can be opened in the Output stream audio configuration options section. (Figure 27)

**Warning:** The audio mode is applied for all outputs of one pipeline!

**Note: **Auto-reconfiguration of audio does not currently work for 2110-30 Audio

*Figure: OUTPUT AUDIO STREAM OPTION DIALOG OPEN*

Passthrough – In this case, all input audio attributes will be kept and set on the audio output, preserving the original channel layout.

Pairs – In this case, the input audio attributes will be changed. The audio/s channels will be separated into up to 8 audio pairs. E.g., for input audio with six channels, the Twenty-One Encoder will produce three audios with stereo sound (2 channels) each.

Quads – Similar mode to Pairs, but in this case, the Twenty-One Encoder creates up to four audio outputs, with four channels each from the main input. Unused channels are silent. E.g., for input audio with six channels, the first output will have four channels with sound, the second audio will have two channels with sound and the remaining two channels will be silent.

*Figure: OUTPUT AUDIO AUTO RECONFIGURATION TYPE SELECTOR*

## 3.8 Manual Low Latency Adjustments

Latency addition is the offset applied to the PTS for reliable transmission to the receiving decoder. The offset is calculated by the overall configuration of the pipeline. The latency addition field allows for the manual adjustment of this offset. This field is available on all MPEG-TS outputs within Twenty-One Encoder.

*Figure: LATENCY ADDITION*

It is recommended to obtain a baseline figure before adjusting the latency addition manually using the Live Transcoder REST API:

Use the following GET command to obtain the pipeline statistics when a pipeline is active: http://hostIPaddress/api/pipeline/0/state

Look for the field “transcodingLatency”

Transcoding Latency is the real time figure of the pipeline, you cannot set the latency addition lower than this.

It is recommended to add 40-50ms+ of the transcoding latency in the Latency Addition field and restart the pipeline

It is best to test using complex content, as this will need more time to encode. If you notice frame drop errors then the Latency Addition will need to be increased.

## 3.9 MPEG-TS output

The output mode (CBR, VBR, CVBR) selects the transmission strategy. For CBR (constant bitrate), Total bitrate field can be set to ensure exact constant network traffic. In VBR (variable bitrate) mode, the transmission corresponds exactly to the amount of data currently produced. With the CVBR (capped VBR), the transmission also corresponds to the amount of data being produced, but won’t exceed the given value in Total bitrate.

**Be aware that this attribute has a direct impact on the resulting quality** – the lower the allowed bitrate, the worse the quality of transmitted data. **This attribute combines the sum of video, audio, and remaining bitrates**.

You can also limit the total video bitrate, which will define which part of the total bitrate is dedicated to video data. In different scenarios, it is necessary to adjust the total bitrate overhead over the video bitrate.

You can choose whether you want PCR synchronization to be included in a video stream, or in a standalone elementary stream within the Transport Stream. It is useful to have it by itself, if you need to have unified PCR synchronization for multiple streams, so in that case it would serve as a “master” PCR for other streams.

With the EBP setting, you define the parameter for Encoder Boundary Point insertion, which is often used in ABR streaming. Following is the section for video preferences where you define the resolution, video bitrate and frame rate.

It is possible to encapsulate the JPEG2000 transport stream in compliance with the standard (TR01) or with Evertz.

## 3.12 MPEG-TS Video passthrough

Live Transcoder has the ability to pass through MPEG-TS without decoding or encoding the video. This is known as video pass through. This is useful for operators that wish to re-encapsulate to/from SRT/UDP/RTP or use Live Transcoder as a one to many device, somewhat like a video IP router.

To configure video passthrough, select “Start Detecting Input”, Twenty-One Encoder will detect what the incoming streams video parameters are. Then select “Set As Input” under the Auto Detected Video Streams. Now when an output stream is created, the video settings will be applied automatically. These will need to be the same as the input to allow for video passthrough. To finish the configuration apply a video bitrate of 0 under the output video configuration settings, and input a total bitrate for the transport stream under the output stream options.

If the output settings for the video are not the same, the following warning will be seen:

*Figure: VIDEO PASSTHROUGH WARNING*

Multiple destinations can be configured to allow a one to many use case. Transport stream encapsulation can also be changed.

## 3.14 Service ID in MPEG-TS output

By clicking on the “Configure service description”, it’s possible to configure several MPEG-TS specifications of the output:

Transport stream ID – A unique 16-bit identifier for this Transport Stream. Used in PAT and SDT tables to distinguish streams within a network. If left empty, a default unspecified ID will be used.

Program number – A 16-bit identifier for the service within the Transport Stream. Called 'service_id' in SDT/PMT and 'program_number' in PAT. If left empty, the value is passed through from the input (if MPEG-TS), or a default unspecified value is used for non-MPEG-TS inputs.

Service name – The name of the service (e.g., TV channel) shown to users. Limited to 24 characters. If left empty, the SDT table will not include service name information.

Service provider name – The name of the company or organization providing this service. Limited to 24 characters. If left empty, the SDT table will not include provider name information.

## 3.15 MPEG-TS video codecs

Available are settings for the chosen video codec that define quality vs. speed difference:

H.262

Profile – Simple, Main, High

GOP size – the distance between two I-frames

B count – number of B-frames between two P-frames

H.264

Coding – CABAC, CAVLC

Profile – Baseline, Main, High

Refresh mode – IDR

Every GOP starts with an IDR I-frame containing the whole picture

GOP size – the distance between two I-frames

B count – number of B-frames between two P-frames

Refresh mode – GDR (intra-refresh)

At the start of each Refresh Period, the encoder inserts Refresh Count P-frames, each refreshing a portion of the picture (one row in NVEnc, one column in the CPU encoder). The remainder of the period consists of regular P-frames. There are no I-frames or B-frames.

Use it to avoid bitrate spikes

Refresh Period – the distance (frames) between two intra refreshes

Refresh Count – number of frames used to refresh the whole picture. Must be equal to or less than Refresh Period. CPU encoder requires Refresh Count to equal Refresh Period.

H.265

Profile – Main10, Main

GOP size – the distance between two I-frames

B count – number of B-frames between two P-frames
