## SRT protocol for reliable data transmission

SRT is a video transport protocol that is intended to be used over unreliable networks such as the public Internet. It amends the packet loss, jitter and fluctuating bandwidth to maintain the high quality of transported video stream. It has end-to-end security mechanisms to protect the transmitted data as well as it is easy to configure to work with network firewalls. If you wish to learn more about the protocol, please visit the official website.

Comprimato Twenty-One Encoder has support for 2 SRT modes of operation – **Listener, Caller**. In other words, this is simply the client–server model. The **Listener unit is configured to connect to its own local IP address that will be** **receiving data** from the (external) Caller unit. Therefore, the **Caller machine is** **configured with the IP address of an external (to it) unit where the data will be** **listened to.**

**Note: **SRT is a connection-oriented communication and therefore cannot use multicast IP delivery.

### Configuring the Input side of SRT transmission

*Figure: Configuring SRT Input Source*

When you select SRT mode:

Listener – Specify the port of the local machine where an external Caller unit will request a connection to start sending data there. If "Any" interface is selected, the SRT connection on the selected port is accepted on all interfaces, thus automatically supporting redundant sources - SRT bonding (for details, see Chapter 4.0.3). Upon selecting a specific interface, a local IP which is listened on, may be also specified.

Caller – IP:port of the remote machine from where the data will be sent

**It is necessary to define the SRT latency variable in milliseconds (ms).** The parameter is the time of intended delay of network packets (a buffer) that is allowed for the connection to slow down the data in transmission in order to provide a time slot to re-send lost UDP packets if they occur. This is what allows the SRT protocol to establish a reliable connection with a focus on removing video disruptions.

Twenty-one encoder also supports the option to enable SRT Encryption. The encryption has to be configured on both ends – caller and listener and uses the same passphrase on both ends (symmetric cryptography). The passphrase needs to have between 10 and 79 characters. Clicking on the icon next to the passphrase you can reveal it.

### Configuring the Output side of SRT transmission

*Figure: Configuring SRT Output*

When you select SRT mode:

Listener – Specify the port of the local machine where an external Caller unit will request a connection so the output which is being configured can start sending data. If "Any" interface is selected, the SRT connection on the selected port is accepted on all interfaces, thus automatically supporting redundant sources - SRT bonding (for details, see Chapter 4.0.3). Upon selecting a specific interface, a local IP which is listened on, may be also specified.

Caller – IP:port of the remote machine where the external unit is waiting to receive data

In addition, **you should define the SRT overhead parameter - an amount of % over** **the Total Bitrate that can be surpassed in order to achieve the lost data recovery.** As a result, the maximum used connection bitrate equals to configured Total Bitrate + SRT overhead. In general, this is the upper limit and connection does not require it for normal operation.

Similarly, to the input configuration, the SRT encryption is supported on the output, too. In output, you have to select which cipher would be used. The available options are AES-128, AES-192, and AES-256. We recommend using AES-256 encryption whenever possible and a strong passphrase where you can use any printable character.

### Stream ID

It’s optional information that you can configure in the SRT Caller mode of operation. The string is sent to the Listener before connecting. If you leave the field empty, the functionality is disabled. The accepted input has up to 512 printable characters.

### SRT bonding (redundancy)

SRT protocol offers a way to ensure maximum reliability by specifying multiple IP paths which the caller and listener are connected through. The most usual case where this possibility is used is a remote connection through the Internet. To use SRT bonding, the Twenty-one encoder would be typically connected to two different networks with Internet access from two different providers (for example optical connection and LTE connection). Such redundancy may be involved either on the Caller or the Listener, or both of them.

For using SRT bonding on the Listener side, no additional configuration is required. By default, the SRT Listener will listen and accept connection for specified port on each connected interface. Please make sure to configure the Sender side to use the same port for all redundant paths.

It is also possible to accept multiple redundant connections coming from one interface in case the redundant paths coming from the Sender would join somewhere along the way, so they come to the Listener from the same interface. That can fit well for cases where the Listener runs with a reliable connection (e.g. in the cloud) as opposed to the Sender which creates the content somewhere in the field.

To use SRT bonding with a Sender, a redundant path may be specified by simply selecting a different interface and IP (see Chapter 4.0.1). Please refer to the configuration of your SRT Listener to match the configured port. If the SRT Listener is the Twenty-One Encoder, the port number of the redundant path must be the same as the port number of the main path.

Twenty-One Encoder currently supports the Broadcast mode of the SRT bonding which ensures 100% seamless protection, as opposed to the Main/Backup mode which is not supported.

## RTMP input

The Twenty-One Encoder supports RTMP streams on input, with a user-provided or generated stream key. RTMPS is not supported yet. The RTMP input listens on port **1935** which is the standard port for RTMP protocol.

To generate the stream, click on the Generate random button to generate a unique stream key. If you prefer, you can manually enter a stream key of your choice. Simply type the desired stream key into the provided input field. Once you have either generated a stream key or entered one manually, it’s essential to copy the RTMP URL to your source device. Twenty-One Encoder generates URLs for every network interface it has.

When the stream key is set the button Start Detecting Input can be clicked to trigger an input stream detection and get video and audio parameters.

The RTMP pipeline supports Auto reconfiguration, allowing for seamless updates to the video and audio input parameters during runtime. This feature ensures continuous processing of the live stream while accommodating changes to the configuration. It can be turned on by checking the Auto Reconfiguration check box and restarting the pipeline.

*Figure: Configuring RTMP input*

## RTMP(S) output

RTMP(S) output in Twenty-one encoder supports both streams with authentication and without it. The protocol is used for video and audio streaming over the Internet and serves as a bridge between a Flash player and a server.

We support the RTMP(S) output only in H.264 video codec and AAC audio codec. It is also worth mentioning that some services (e.g., YouTube) don’t accept RTMP(S) streams without any audio.

Comprimato Twenty-One Encoder is also mentioned as a Qualified Encoder by Akamai in a list that you can find here.

### Configuring RTMP(S) output

To set up an RTMP(S) output stream, the first thing you need to do is to select Output stream options, Stream type as RTMP(S) and just beneath it, fill out the URL address (example of a URL of a stream with authentication: rtmp://user:pass@IPADDRESS/application/stream-key). If you have provided a valid URL, Twenty-one encoder will produce an output stream upon starting the configured pipeline using RTMP(S).

*Figure: Configuring RTMP(S)*

If you want to use a hostname instead of an IP address in the URL, you need to set an appropriate DNS server and if it’s in the external network, the set Default route needs to have access to the Internet (both can be done in the Setup, more about that in the section 2.1).

## HLS

HTTP Live Streaming is an adaptive bitrate streaming protocol. It is one of the most popular streaming formats.

As of now, we support only HLS on the input, with a few limitations:

users are unable to select a particular time in the VOD HLS stream, it starts processing VOD from the start, unless it’s also live. Then Twenty-one encoder processes the newest part of the stream to follow with the live transmission,

users are unable to select a particular stream from the HLS ladder if more are present. Twenty-one encoder automatically chooses the one with the highest quality for conversion,

supported video codecs within HLS are H.264 and H.265, whereas, for audio, it is EAC3, AC3 and AAC

Twenty-one encoder does not verify certificate validity when using an HTTPS stream.

### Configuring HLS input

To set up an HLS input stream, the first thing you need to do is to navigate to the Input stream source and set the type as HLS and just beneath it, fill out the URL address (example of a stream URL: http://source.example/file.m3u8). If you have provided a valid URL, Twenty-one encoder will ingest the input stream and autodetect contents which you then can confirm and later set an output stream.

You also must configure the Buffer size. We recommend using a value 7 to 10 times higher than the length of the HLS segment. At start, you will have to wait the set amount for the video to buffer before you can see any output. Make sure that the buffer is not too short (e. g. single-digit duration of it), since it may cause buffer due to an unstable network.

*Figure: HLS configuration*

If you want to use a hostname instead of an IP address in the URL, you need to set an appropriate DNS server and if it’s in the external network, the set Default route needs to have access to the Internet (both can be done in the Setup, more about that in the section 2.1)

### Configuring HLS output

If you set HLS on output, Twenty-One Encoder will start behaving like a server and you can connect to that server. UI automatically shows all URL addresses where you can find the stream.

## NDI

Network Device Interface (NDI) is a high-performance standard that allows anyone to use real-time, ultra-low latency video on existing IP video networks.

NDI is part of the Twenty-One Encoder and here is how to use it.

### Configuring NDI input

In the detected NDI list you can see all available NDI sources on the local network.

*Figure: NDI configuration*

If you want to start using a listed NDI input, just select it in the list. It will be set to the Selected NDI field. In some cases, you may want to set the NDI source name manually (e.g., when it's not yet running). You can do so by filling in the Selected NDI field yourself. Please make sure the name is filled in exactly.

*Figure: NDI detecting input*

Then you can **click the Start Detecting Input** button to get the NDI feed configuration.

### NDI sync options

There are three different sync modes for NDI input within Live Transcoder:

Free-run – Synchronises video to the audio. This may drop video frames in order to resync.

Frame sync – Synchronises both video and audio using the local clock. Audio is re-sampled and video is duplicated/dropped if needed.

AV sync – Feature from NDI advanced SDK. Currently experimental, testing is recommended before using.

### Configuring NDI input in non-default NDI group

NDI allows sources to be part of so-called NDI groups. A source can be a part of any number of groups and is findable only if the corresponding group is specified.

To specify an NDI group to find a source in or get a list of detected sources in the group, please allow Advanced NDI options and enter the group name into the corresponding field. Once the field is filled, the list of detected sources is updated with the sources found in the selected group.

The NDI group name field can also contain a comma-separated list of groups, all of them will be used to search for the specified NDI source and to list the detected sources. The default group name is 'public' so if the Advanced NDI options are not allowed or the NDI group name field isn't set, the sources are detected in the 'public' group. (edited)

### NDI discovery server

If you are trying to point the NDI output of Twenty-One Encoder to one or more NDI Discovery Servers, you have to change the appropriate parameter in the .ini configuration file in the Advanced setup:

Find and replace the line containing **NDI_DISCOVERY=** with the destination IP(s) of the NDI Discovery server(s). Use commas to separate multiple servers, e.g.,

**NDI_DISCOVERY=192.168.101.55,192.168.101.56**

Warning: When the NDI Discovery Server is set, the NDI output (sender) doesn’t propagate itself using mDNS, just to the Discovery Server. NDI input (receiver) receives traffic from both mDNS and Discovery Server. This is how NDI works and you can’t change it.

### I can’t discover Twenty-one encoder’s NDI output

Check that your network interface is configured for “Input/Output” or “Input/Output and Administration” in the UI Setup. This effectively configures the firewall in the OS.

### I can discover Twenty-one encoder’s NDI output, but it doesn’t play

Since NDI 5, NDI uses RUDP (Reliable UDP) as a communication protocol.  Some network cards have problems with RUDP. In the NDI Access Manager application on your receiver device, you can set the “Receiver Mode” to “Single-TCP” and restart both the NDI output pipeline in Twenty-one encoder and the NDI application on your receiver device.

## Auto reconfiguration

Twenty-One Encoder allows the auto reconfiguration of the pipelines. It means it can automatically restart a pipeline when it detects a change in the parameters of the input video or audio. The video outputs are kept unchanged, the behaviour of audio outputs is described later. The auto reconfiguration can be turned on by a checkbox. This option is not available with ST 2110 input and will instead be controlled by NMOS IS-05 (Figure 48)

**Figure 48 auto reconfiguration enabling**

### Auto reconfiguration of Audio outputs

There are three modes of output audio channel layout. Passthrough, Pairs and Quads. They can be selected in the dialog window, which can be opened in the Output stream audio configuration options section. (Figure 49)

**Warning:** The audio mode is applied for all outputs of one pipeline!

**Figure 49 output audio stream option dialog open**

Passthrough – In this case, all input audio attributes will be kept and set on the audio output, preserving the original channel layout.

Pairs – In this case, the input audio attributes will be changed. The audio/s channels will be separated into up to 8 audio pairs. E.g., for input audio with six channels, the Twenty-One Encoder will produce three audios with stereo sound (2 channels) each.

Quads – Similar mode to Pairs, but in this case, the Twenty-One Encoder creates up to four audio outputs, with four channels each from the main input. Unused channels are silent. E.g., for input audio with six channels, the first output will have four channels with sound, the second audio will have two channels with sound and the remaining two channels will be silent.

**Figure 50 output audio auto reconfiguration type selector**

## Timestamp SEI support (MISB ST 0604.6)

Twenty-One Encoder supports timestamps SEI messages according to the MISB ST 0604.6 standard. Timestamp values are obtained from the local time of the machine Twenty-One Encoder runs at and they are inserted as SEI into the output stream. The time of the machine is supposed to be synchronized using the NTP.

The timestamps SEI messages are supported for the following inputs:

MPEG-TS

NDI

and can be inserted into H.264 or H.265 video outputs.

By default, the insertion of timestamps SEI into the output streams is turned off. To turn it on the following settings.ini variables can be used:

; timestamp SEI messages (MISB 0604)

; --------------------------------------------------------------

; enable/disable adding timecode SEI messages for H.264 nvenc (default: disabled)

; NVENC_H264_ADD_TIMESTAMPS = 1

; enable/disable adding timecode SEI messages for H.264 cpu encoder (default: disabled)

; X264_ADD_TIMESTAMPS = 1

; enable/disable adding timecode SEI messages for H.265 nvenc (default: disabled)

; NVENC_HEVC_ADD_TIMESTAMPS = 1

Moreover, a custom timestamp identifier can be provided for the H.265 streams. If this setting is omitted (default) the standard identifier specified by the MISB ST 0604.6 is used.

; timestamp identifier to be used as the uuid_iso_iec_11578 of the user data SEI message

; ommit to use the standard one specified in MISB ST 0604 (works for H.265 nvenc only!)

; NVENC_HEVC_TIMESTAMPS_IDENTIFIER = cf848278-ee23-306c-9265-e8fef22fb8b8

### Configuring PTP

PTP is configured from factory. The default PTP domain configured is: 127

There is status for PTP synchronization within the UI, under System Information:

PTP Synchronization: Is enabled by default.

A nominated grandmaster must be present for synchronization.

PTP Grandmaster ID is the MAC address of the PTP Grandmaster network interface.

PTP domain is the domain for interacting set of PTP clocks.

PTP service can be configured in the following way:

```bash
# Change Domain and other important numbers

sudo nano /etc/ptp4l.0.conf
sudo nano /etc/ptp4l.1.conf

# change domain in parameter -n <number>

sudo nano /etc/systemd/system/phc2sys-0.service
sudo nano /etc/systemd/system/phc2sys-1.service

# after every change of one service this must be called (or after bunch of changes):
sudo systemctl daemon-reload

# Check the status of each service for synchronization
sudo systemctl restart phc2sys-0.service
sudo systemctl restart phc2sys-1.service
sudo systemctl restart ptp4l-0.service
sudo systemctl restart ptp4l-1.service
```

### Registering Twenty-One Encoder within NMOS registry

When Twenty-One Encoder is connected to a network containing NMOS registry that is compatible with IS-04 Distributed Registry, it will automatically register in the following steps:

Ensure interface eno1np0 link is up (labelled LAN 1 on the server backplate) and IP address is configured, this is the network interface that NMOS configuration is enabled on by default.

Self-register as NMOS node, this will be labelled as the hostname of the machine which is set in the Setup dialogue.

Any existing pipelines with ST 2110 inputs configured will be registered as NMOS devices with the pipeline’s name.

Video and audio receivers will be registered for each pipeline: 1x 2110-20 video receiver and 8x 2110-30 audio receivers.

If you wish for NMOS configuration to be made available on a different network interface, please use the following procedure to reconfigure:

SSH to server

Ifconfig (check the name of the network interface you would like to use)

sudo ostree admin unlock

cd etc/transcoder/transcoder

nano ui.cfg

change: nmos.nic=*NIC name*

save file

reboot server

### ST 2110 Troubleshooting

The status will not immediately reflect changes to the source. All changes to the video source must be updated in the SDP file.

A good example of this is if the frame rate of the source changes from 25fps to 29.97fps. This is reflected on the sender side of the NMOS registry but until the updated SDP file is sent to the receiver; Twenty-One Encoder pipeline in this case. The output would show a warning as the pipeline is still configured to receive 25fps:
